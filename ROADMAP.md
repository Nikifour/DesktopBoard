## Что закладываем уже сейчас

- Стабильные уникальные `id` у карточек и связей
- Миграции формата состояния
- Локальное хранилище файлов через абстракцию ассетов, а не через внешние пути
- Внутренний реестр типов карточек: `kind`, иконка, заголовок, дефолтные размеры, дефолтные цвета, правила быстрого создания
- Внутренний command/action layer для создания карточек и запуска действий из UI без публичного внешнего API

## API / интеграции

- [x] Сейчас заложить только внутренний API приложения:
  - реестр типов карточек;
  - единые команды `create / duplicate / delete / focus / connect`;
  - изолированные адаптеры для файлов, web-элементов и будущих интеграций
- [ ] Публичный API, плагины и внешние сервисы делать позже, после стабилизации:
  - модели состояния;
  - undo/redo;
  - нескольких досок / пространств;
  - жизненного цикла приложения и автообновлений

## Что пока осознанно НЕ делаем

- Авторизацию
- Серверную синхронизацию
- Платежи
- Совместное редактирование

Причина простая: это отдельный продуктовый и инфраструктурный слой. Его лучше добавлять после стабилизации локального UX и модели данных.
